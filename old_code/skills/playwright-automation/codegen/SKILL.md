# Code Generation

## Step 1 — Show Step Review First
ALWAYS show recorded steps before generating.
Never generate without user review.

  "Here are your recorded steps:
   1. Navigate to [url]
   2. Fill [element] with [value]
   3. Click [element]
   4. Assert [element] is visible

   Want to remove, reorder, or add anything
   before I generate the script? [Yes] [No]"

Wait for user confirmation.
Only generate after user approves.

## Step 2 — Generate the File

File naming:
  .hermes/output/[date]-[test-name].spec.ts
  Example: 2026-04-30-login-flow.spec.ts

Always use this exact structure:

// ============================================
// Generated by Hermes Automation Co-pilot
// Session: [ISO timestamp]
// App: [URL from browser_get_state]
// Test: [session name]
// ============================================

import { test, expect } from '@playwright/test'

// === LOCATORS ===
// All locators defined here for easy maintenance
// Update this section when app changes

const emailInput = page.getByLabel('Email')
const passwordInput = page.getByLabel('Password')
const loginButton = page.getByRole('button',
  { name: 'Login' })
const dashboardHeading = page.getByRole(
  'heading', { name: 'Dashboard' })

// ⚠ Fragile locator — no stable attributes found
// Consider asking developer to add data-testid
const submitBtn = page.locator(
  '//form/div[3]/button')

// === TEST ===
test('[test name]', async ({ page }) => {
  await page.goto(
    process.env.BASE_URL ??
    'https://app.example.com')

  await emailInput.fill(
    process.env.TEST_EMAIL ??
    'test@example.com')

  await passwordInput.fill(
    process.env.TEST_PASSWORD ?? '')

  await loginButton.click()

  await expect(dashboardHeading
    ).toBeVisible()
})

## Locator Variable Naming Rules
button   → loginButton, submitButton
input    → emailInput, passwordInput
link     → dashboardLink, logoutLink
heading  → pageHeading, sectionTitle
modal    → confirmModal, errorModal
text     → successMessage, errorText

Never use:
  btn1, input1, el1, loc1

## Code Rules
1. ALL locators go at TOP in LOCATORS section
2. Use most stable locator found
3. Add warning comment for fragile locators
4. Use process.env for ALL secrets
5. Never hardcode passwords or tokens
6. Add waitForLoadState after navigation
7. Never use page.waitForTimeout() — no sleep
8. All assertions use expect() not console.log
9. TypeScript only — no Python
10. Add test tags if user specified any:
    test('login flow @smoke @auth', ...)

## After Generating
1. Write file to .hermes/output/
2. Tell user: "Script saved to: [path]"
3. Show preview of first 20 lines
4. Ask: "Want to run this test now?"
   If yes: run using terminal_tool:
   npx playwright test [filepath]

## Handling Faker-Generated Data in Output
When steps used Faker to generate test data,
generated code must make this explicit.

DO NOT hardcode the Faker value that was used:
  // WRONG:
  await emailInput.fill('john.doe@example.com')
  // This is brittle — same email every run

DO use Faker in generated code:
  // CORRECT:
  import { faker } from '@faker-js/faker'
  const testEmail = faker.internet.email()
  const testName  = faker.person.fullName()
  const testPhone = faker.phone.number()

  await emailInput.fill(testEmail)
  await nameInput.fill(testName)

Full structure when Faker is used:

// ============================================
// Generated by Hermes Automation Co-pilot
// Session: [timestamp]
// App: [URL]
// Test: [test name]
// ============================================

import { test, expect } from '@playwright/test'
import { faker } from '@faker-js/faker'

// === TEST DATA ===
// Generated fresh each run
const testEmail = faker.internet.email()
const testName  = faker.person.fullName()
const testPhone = faker.phone.number()

// === LOCATORS ===
const emailInput = page.getByLabel('Email')
const nameInput  = page.getByLabel('Full name')

// === TEST ===
test('[test name]', async ({ page }) => {
  await page.goto(process.env.BASE_URL ?? '')
  await emailInput.fill(testEmail)
  await nameInput.fill(testName)
})

Install Faker if not in project:
  Tell user: "Run: npm install @faker-js/faker"

## Handling Multi-Tab Flows in Output
When recorded steps involved new tabs,
generated code must handle tab context correctly.

Structure for multi-tab test:

// === LOCATORS — Main Tab ===
const openReportLink = page.getByRole(
  'link', { name: 'Open Report' })

// === TEST ===
test('[test name]', async ({
  page, context
}) => {
  // Steps on main tab
  await page.goto(process.env.BASE_URL ?? '')

  // Step that opens new tab
  const [newTab] = await Promise.all([
    context.waitForEvent('page'),
    openReportLink.click()
  ])
  await newTab.waitForLoadState()

  // === LOCATORS — New Tab ===
  const reportHeading = newTab.getByRole(
    'heading', { name: 'Report' })

  // Steps on new tab
  await expect(reportHeading).toBeVisible()
  await expect(newTab).toHaveURL(
    /.*\/reports\/.*/
  )

  // Close new tab
  await newTab.close()

  // Back on main tab — continue steps
  await expect(page).toHaveURL(
    process.env.BASE_URL ?? ''
  )
})

Key rules for multi-tab output:
  Always add context to test parameters
  Always use Promise.all for tab capture
  Label locators by which tab they belong to
  Always await waitForLoadState on new tab
  Always close new tab when done with it

## Handling Soft Assertions in Output
When user recorded multiple assertions
that should all be checked even if one fails:

import { test, expect } from '@playwright/test'

test('[test name]', async ({ page }) => {
  await page.goto(process.env.BASE_URL ?? '')

  // Soft assertions — all run even if one fails
  await expect.soft(
    page.getByLabel('Email')
  ).toBeVisible()
  
  await expect.soft(
    page.getByLabel('Password')
  ).toBeVisible()
  
  await expect.soft(
    page.getByRole('button', { name: 'Login' })
  ).toBeEnabled()

  // Hard assertion — stops test if fails
  await expect(page).toHaveTitle(
    'Login — My App')
})

When to use soft assertions:
  User says "check all these fields"
  User says "verify the whole form"
  Multiple related assertions in one step
  Checking a page has all required elements

When to use hard assertions:
  Critical path step (login, navigation)
  One specific important check
  Assertion that must pass before continuing

## Handling Auth State in Output
When session used storage state:

import { test, expect } from '@playwright/test'

// Use saved auth state — skip login
test.use({
  storageState: '.hermes/auth/storageState.json'
})

test('[test name]', async ({ page }) => {
  // Go directly to authenticated page
  // No login steps needed
  await page.goto(
    process.env.BASE_URL ?? '' + '/dashboard'
  )
  await expect(
    page.getByRole('heading',
      { name: 'Dashboard' })
  ).toBeVisible()
})

Add this note as comment in generated file:
  // Auth: uses saved storage state
  // To refresh: run auth setup script first
  // or say 'save storage state' in Hermes
